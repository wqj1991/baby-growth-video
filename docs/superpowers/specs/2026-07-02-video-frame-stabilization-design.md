# 视频抽帧能力稳定化设计（方案B）

## 1. 背景与目标

当前项目的视频抽帧链路已具备基础能力（设置弹窗、抽帧命令、内嵌播放器、加入待选区），但存在“功能链路可触达却不稳定”的问题：

- 按间隔抽帧存在前后端断链风险（前端可调用但后端命令未完整暴露）
- 抽帧入口状态与文案不统一，用户难以预期当前阶段
- 抽帧成功/失败后的计数与按钮状态可能不一致
- 重复点击可能导致重复请求，增加失败率和状态混乱概率

本次目标：在不引入新流程（不新增抽帧结果弹窗）的前提下，完成现有主链稳定化。

目标主链：设置弹窗 -> 抽帧命令 -> 加载临时帧 -> 内嵌播放器 -> 加入待选区

### 1.1 本次必须达成

1. 按数量、按间隔两种抽帧模式都可用
2. 抽帧后可稳定进入内嵌播放器并处理临时帧
3. 按钮文案、加载态、错误提示一致且可预期
4. 抽帧计数与界面状态保持一致

### 1.2 本次不包含

1. 不引入新的抽帧结果弹窗流程
2. 不改待选区数据模型
3. 不进行大规模页面重构

---

## 2. 方案对比与选型

### 2.1 方案A：极小修补

只修后端命令暴露与前端最小文案。

- 优点：改动最小，交付最快
- 缺点：状态一致性与交互稳定性不足

### 2.2 方案B：稳定化闭环（选定）

在方案A基础上补齐状态一致性和交互规范，不改变主流程。

- 优点：满足“功能完整 + 体验一致”
- 缺点：改动面略大于方案A，但仍可控

### 2.3 方案C：稳定化 + 轻度整理

在方案B基础上进一步抽离页面逻辑。

- 优点：长期维护更优
- 缺点：超出本轮最小改动范围

结论：采用方案B。

---

## 3. 架构边界与模块职责

### 3.1 设计原则

1. 不新增产品路径，保持现有“设置弹窗 -> 内嵌播放器”
2. 只补断链，不做无关重构
3. 以临时帧为中间态，待选区为落点，不引入并行状态源
4. 抽帧入口行为统一，按钮状态由同一组规则驱动

### 3.2 涉及模块

- 后端命令注册层：确保两种抽帧命令都可被前端调用
- 前端命令封装层：统一调用参数与错误传递
- 页面编排层：管理弹窗、命令调用、播放器打开、计数刷新
- 播放器层：消费临时帧并支持加入待选区

### 3.3 不变边界

- 不新增组件级新流程
- 不改现有待选区业务模型
- 不引入新依赖

---

## 4. 关键流程设计

### 4.1 主流程

1. 用户点击视频卡片按钮
2. 若无临时帧：打开设置弹窗，选择数量或间隔并确认
3. 进入 extracting 状态并发起抽帧命令
4. 抽帧成功后加载临时帧，刷新数量，进入 ready 状态
5. 自动打开内嵌播放器查看并处理临时帧
6. 用户可将临时帧加入待选区

### 4.2 查看流程

1. 若已有临时帧，按钮显示“查看(n)”
2. 点击后跳过抽帧，直接加载并打开内嵌播放器

### 4.3 清理流程

1. 放弃或退出时清理对应临时帧
2. 同步刷新计数，按钮回退到“截取画面”或更新后的“查看(n)”

---

## 5. 状态模型与一致性规则

### 5.1 抽帧入口状态机

- idle：默认可操作
- extracting：请求进行中，禁用重复触发
- ready：已有临时帧，支持查看
- failed：请求失败，可重试

### 5.2 状态转换规则

1. idle -> extracting：设置弹窗确认后
2. extracting -> ready：命令成功且临时帧加载成功
3. extracting -> failed：命令失败或加载失败
4. failed -> extracting：用户再次发起
5. ready -> idle：临时帧清空后

### 5.3 计数一致性规则

1. n 的来源统一为当前视频临时帧数
2. 只在成功路径刷新 n
3. 清理后立即回写 n
4. 失败路径不修改 n

---

## 6. 交互与文案规范

### 6.1 按钮文案

- 无帧：截取画面
- 有帧：查看(n)
- 处理中：抽帧中...

### 6.2 反馈规范

- 成功：按现有成功提示体系展示
- 失败：统一 toast 文案为“抽帧失败，请重试”
- 不再混用 alert 与静默失败

### 6.3 并发与防重复

1. 同一视频 extracting 期间拒绝重复触发
2. 快速连点只接受首个请求
3. 不影响同卡片“播放”能力

---

## 7. 错误处理与回退策略

### 7.1 错误分类

1. 参数错误：前端校验并阻止发起
2. 命令调用错误：统一系统错误提示
3. 抽帧执行失败：恢复按钮、允许重试
4. 临时帧加载失败：不进入误导性空播放器态

### 7.2 回退原则

1. 任意失败都必须退出 extracting
2. 失败不污染计数状态
3. 不自动删除已有可用临时帧
4. 用户可直接重试或进入查看流程

## 9. 验证策略与验收标准

### 9.1 验证策略

1. 后端：cargo check 通过
2. 前端：pnpm build 通过
3. 手工链路验证覆盖数量模式、间隔模式、失败重试、查看流程、计数回写

### 9.2 验收用例

1. 数量模式抽帧成功并显示查看(n)
2. 间隔模式抽帧成功并显示查看(n)
3. extracting 时重复点击无二次请求
4. 抽帧失败后可再次成功发起
5. 加入待选区后当前页状态与计数正常

### 9.3 Done 定义

1. 数量与间隔模式均闭环可用
2. 文案、加载、错误反馈一致
3. 计数与状态无明显错位
4. 构建与检查通过

---

## 10. 风险与缓解

1. 风险：后端命令签名与前端参数名不一致
   - 缓解：统一参数命名并在本地链路验证
2. 风险：局部状态改动影响其他视频卡片
   - 缓解：将抽帧锁粒度限制到当前视频
3. 风险：临时帧清理与计数刷新竞态
   - 缓解：清理后立即重新加载该视频临时帧并回写计数

---

## 11. 结论

本设计在保持现有产品路径不变的前提下，以最小改动达成“功能完整 + 体验一致”的视频抽帧稳定化目标。设计范围清晰、验证标准明确，可直接进入 implementation planning 阶段。
